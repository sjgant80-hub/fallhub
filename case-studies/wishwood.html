Wishwood Glamping · the reference operator.

Wishwood is a 4-unit off-grid glamping site near Canterbury run by Chrissy, a nine-year hosting veteran with a 4.81★ average across 779 reviews. It's the reference deployment of FallHub — everything you see there is what you get when you fork.

4 units Kent, UK Live since 2026-06 wishwood.co.uk

What was there before

Chrissy was running the standard off-grid-glamping tech stack: Hostaway for channel management (£79/month), a separate booking widget on her Wix site, four browser tabs open every morning (Airbnb, Booking.com, WhatsApp, email), and a personal notes doc for tracking guest comms she couldn't lose. Every time a channel manager pushed a rate-change or a booking, she had to double-check it hadn't broken something else.

Total tool spend: ~£948/year. Total attention spend on inbox unification: about 90 minutes per morning, seven days a week.

"The tech was supposed to save me time. It gave me four tabs and a spreadsheet." — Chrissy, before FallHub

Deployment story

  • AI-Native Solutions forked the FallHub template into a Wishwood-branded repo
  • Chrissy filled in the credentials.html checklist over two evenings (Stripe, Twilio for WhatsApp, Resend for email, Cloudflare for the Worker)
  • AI-Native Solutions filled the config JSON with the 4 cabins, their pricing, iCal URLs, and Wishwood's woodland palette
  • Cloudflare Worker deployed in one command with `npx wrangler deploy`
  • DNS switched over during quiet season (2 hours downtime, planned)
  • Hostaway subscription cancelled the day after go-live
  • Full end-to-end test: a real booking arrived via Airbnb within 48 hours — hub.html showed it, AI drafted a welcome message, Chrissy approved

Elapsed time from fork to first live booking: about 5 days. Most of that time was waiting for Twilio to approve the WhatsApp sender.

What's still ahead for Wishwood

The Roost integration (a founding-host cooperative marketplace at roost) is the next wedge — same FallHub codebase, additional channel added to the config. Add-on comms channels (Instagram DM, Facebook Messenger) are dropping in one at a time as guest volume from each grows. Neither required a new codebase — just config edits.

What this proves about FallHub

  • Config beats code · reskinning for a new operator vertical (glamping → B&B → treehouse) is a config edit, not a rewrite
  • The unified inbox pattern generalises · adding a new channel is one config entry + one webhook
  • Free-tier Cloudflare Workers comfortably handle a real small operator's traffic
  • The AI draft pattern preserves operator voice · Chrissy's replies still sound like Chrissy
  • MIT source + Cloudflare + GitHub Pages means the operator survives FallHub disappearing
